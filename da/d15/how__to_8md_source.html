2 BTstack implements a set of basic Bluetooth protocols. To make use of
3 these to connect to other devices or to provide own services, BTstack
4 has to be properly configured during application startup.
5 
6 In the following, we provide an overview of the memory management, the
7 run loop, and services that are necessary to setup BTstack. From the
8 point when the run loop is executed, the application runs as a finite
9 state machine, which processes events received from BTstack. BTstack
10 groups events logically and provides them over packet handlers, of which
11 an overview is provided here. Finally, we describe the RFCOMM
12 credit-based flow-control, which may be necessary for
13 resource-constraint devices.

57 ## Run loop {#sec:runLoopHowTo}
58 
59 BTstack uses a run loop to handle incoming data and to schedule work.
60 The run loop handles events from two different types of sources: data
61 sources and timers. Data sources represent communication interfaces like
62 an UART or an USB driver. Timers are used by BTstack to implement
63 various Bluetooth-related timeouts. They can also be used to handle
64 periodic events.
65 
66 Data sources and timers are represented by the *data_source_t* and
67 *timer_source_t* structs respectively. Each of these structs contain a
68 linked list node and a pointer to a callback function. All active timers
69 and data sources are kept in link lists. While the list of data sources
70 is unsorted, the timers are sorted by expiration timeout for efficient
71 processing.
72 
73 The complete run loop cycle looks like this: first, the callback
74 function of all registered data sources are called in a round robin way.
75 Then, the callback functions of timers that are ready are executed.
76 Finally, it will be checked if another run loop iteration has been
77 requested by an interrupt handler. If not, the run loop will put the MCU
78 into sleep mode.
79 
80 Incoming data over the UART, USB, or timer ticks will generate an
81 interrupt and wake up the microcontroller. In order to avoid the
82 situation where a data source becomes ready just before the run loop
83 enters sleep mode, an interrupt-driven data source has to call the
84 *embedded_trigger* function. The call to *embedded_trigger* sets an
85 internal flag that is checked in the critical section just before
86 entering sleep mode.
87 
88 Timers are single shot: a timer will be removed from the timer list
89 before its event handler callback is executed. If you need a periodic
90 timer, you can re-register the same timer source in the callback
91 function, as shown in Listing [PeriodicTimerHandler]. Note that BTstack
92 expects to get called periodically to keep its time, see Section
93 [on time abstraction](#sec:timeAbstractionPorting) for more on the
94 tick hardware abstraction.
95 
96 The run loop is set up by calling *run_loop_init* function for
97 embedded systems:
98 
99 <!-- -->
100 
101  run_loop_init(RUN_LOOP_EMBEDDED);
102 
103 The Run loop API is provided [here](appendix/apis/#sec:runLoopAPIAppendix). To
104 enable the use of timers, make sure that you defined HAVE_TICK in the
105 config file.
106 
107 In your code, you’ll have to configure the run loop before you start it
108 as shown in Listing [listing:btstackInit]. The application can register
109 data sources as well as timers, e.g., periodical sampling of sensors, or
110 communication over the UART.

113 ## BTstack initialization {#sec:btstackInitializationHowTo}
114 
115 To initialize BTstack you need to [initialize the memory](#sec:memoryConfigurationHowTo)
116 and [the run loop](#sec:runLoopHowTo) respectively, then setup HCI and all needed higher
117 level protocols.
118 
119 The HCI initialization has to adapt BTstack to the used platform and
120 requires four arguments. These are:
121 
122 - *Bluetooth hardware control*: The Bluetooth hardware control API can
123  provide the HCI layer with a custom initialization script, a
124  vendor-specific baud rate change command, and system power
125  notifications. It is also used to control the power mode of the
126  Bluetooth module, i.e., turning it on/off and putting to sleep. In
127  addition, it provides an error handler *hw_error* that is called
128  when a Hardware Error is reported by the Bluetooth module. The
129  callback allows for persistent logging or signaling of this failure.
130 
131  Overall, the struct *bt_control_t* encapsulates common
132  functionality that is not covered by the Bluetooth specification. As
133  an example, the *bt_con-trol_cc256x_in-stance* function returns a
134  pointer to a control struct suitable for the CC256x chipset.
135 
136 <!-- -->
137 
138  bt_control_t * control = bt_control_cc256x_instance();
139 
140 - *HCI Transport implementation*: On embedded systems, a Bluetooth
141  module can be connected via USB or an UART port. BTstack implements
142  two UART based protocols: HCI UART Transport Layer (H4) and H4 with
143  eHCILL support, a lightweight low-power variant by Texas
144  Instruments. These are accessed by linking the appropriate file
145  [src/hci_transport_h4_dma.c]() resp. [src/hci_transport_h4_ehcill_dma.c]()
146  and then getting a pointer to HCI Transport implementation.
147  For more information on adapting HCI Transport to different
148  environments, see [here](porting/#sec:hciTransportPorting).
149 
150 <!-- -->
151 
152  hci_transport_t * transport = hci_transport_h4_dma_instance();
153 
154 - *HCI Transport configuration*: As the configuration of the UART used
155  in the H4 transport interface are not standardized, it has to be
156  provided by the main application to BTstack. In addition to the
157  initial UART baud rate, the main baud rate can be specified. The HCI
158  layer of BTstack will change the init baud rate to the main one
159  after the basic setup of the Bluetooth module. A baud rate change
160  has to be done in a coordinated way at both HCI and hardware level.
161  First, the HCI command to change the baud rate is sent, then it is
162  necessary to wait for the confirmation event from the Bluetooth
163  module. Only now, can the UART baud rate changed. As an example, the
164  CC256x has to be initialized at 115200 and can then be used at
165  higher speeds.
166 
167 <!-- -->
168 
169  hci_uart_config_t* config = hci_uart_config_cc256x_instance();
170 
171 - *Persistent storage* - specifies where to persist data like link
172  keys or remote device names. This commonly requires platform
173  specific code to access the MCU’s EEPROM of Flash storage. For the
174  first steps, BTstack provides a (non) persistent store in memory.
175  For more see [here](porting/#sec:persistentStoragePorting).
176 
177 <!-- -->
178 
179  remote_device_db_t * remote_db = &remote_device_db_memory;
180 
181 After these are ready, HCI is initialized like this:
182 
183 <!-- -->
184  hci_init(transport, config, control, remote_db);
185 
186 The higher layers only rely on BTstack and are initialized by calling
187 the respective *\*_init* function. These init functions register
188 themselves with the underlying layer. In addition, the application can
189 register packet handlers to get events and data as explained in the
190 following section.

193 ## Services {#sec:servicesHowTo}
194 
195 One important construct of BTstack is *service*. A service represents a
196 server side component that handles incoming connections. So far, BTstack
197 provides L2CAP and RFCOMM services. An L2CAP service handles incoming
198 connections for an L2CAP channel and is registered with its protocol
199 service multiplexer ID (PSM). Similarly, an RFCOMM service handles
200 incoming RFCOMM connections and is registered with the RFCOMM channel
201 ID. Outgoing connections require no special registration, they are
202 created by the application when needed.

206 ## Where to get data - packet handlers {#sec:packetHandlersHowTo}
207 
208 
209 After the hardware and BTstack are set up, the run loop is entered. From
210 now on everything is event driven. The application calls BTstack
211 functions, which in turn may send commands to the Bluetooth module. The
212 resulting events are delivered back to the application. Instead of
213 writing a single callback handler for each possible event (as it is done
214 in some other Bluetooth stacks), BTstack groups events logically and
215 provides them over a single generic interface. Appendix
216 [Events and Errors](generated/appendix/#sec:eventsAndErrorsAppendix)
217 summarizes the parameters and event
218 codes of L2CAP and RFCOMM events, as well as possible errors and the
219 corresponding error codes.
220 
221 Here is summarized list of packet handlers that an application might
222 use:
223 
224 - HCI packet handler - handles HCI and general BTstack events if L2CAP
225  is not used (rare case).
226 
227 - L2CAP packet handler - handles HCI and general BTstack events.
228 
229 - L2CAP service packet handler - handles incoming L2CAP connections,
230  i.e., channels initiated by the remote.
231 
232 - L2CAP channel packet handler - handles outgoing L2CAP connections,
233  i.e., channels initiated internally.
234 
235 - RFCOMM packet handler - handles RFCOMM incoming/outgoing events and
236  data.
237 
238 These handlers are registered with the functions listed in Table
239 {@tbl:registeringFunction}.
240 
241  ------------------------------ --------------------------------------
242  Packet Handler Registering Function
243  HCI packet handler *hci_register_packet_handler*
244  L2CAP packet handler *l2cap_register_packet_handler*
245  L2CAP service packet handler *l2cap_register_service_internal*
246  L2CAP channel packet handler *l2cap_create_channel_internal*
247  RFCOMM packet handler *rfcomm_register_packet_handler*
248  ------------------------------ --------------------------------------
249 
250 Table: Functions for registering packet handlers. {#tbl:registeringFunction}
251 
252 HCI and general BTstack events are delivered to the packet handler
253 specified by *l2cap_register_packet_handler* function, or
254 *hci_register_packet_handler*, if L2CAP is not used. In L2CAP,
255 BTstack discriminates incoming and outgoing connections, i.e., event and
256 data packets are delivered to different packet handlers. Outgoing
257 connections are used access remote services, incoming connections are
258 used to provide services. For incoming connections, the packet handler
259 specified by *l2cap_register_service* is used. For outgoing
260 connections, the handler provided by *l2cap_create_channel_internal*
261 is used. Currently, RFCOMM provides only a single packet handler
262 specified by *rfcomm_register_packet_handler* for all RFCOMM
263 connections, but this will be fixed in the next API overhaul.
264 
265 The application can register a single shared packet handler for all
266 protocols and services, or use separate packet handlers for each
267 protocol layer and service. A shared packet handler is often used for
268 stack initialization and connection management.
269 
270 Separate packet handlers can be used for each L2CAP service and outgoing
271 connection. For example, to connect with a Bluetooth HID keyboard, your
272 application could use three packet handlers: one to handle HCI events
273 during discovery of a keyboard registered by
274 *l2cap_register_packet_handler*; one that will be registered to an
275 outgoing L2CAP channel to connect to keyboard and to receive keyboard
276 data registered by *l2cap_create_channel_internal*; after that
277 keyboard can reconnect by itself. For this, you need to register L2CAP
278 services for the HID Control and HID Interrupt PSMs using
279 *l2cap_register_service_internal*. In this call, you’ll also specify
280 a packet handler to accept and receive keyboard data.

284 ## Bluetooth HCI Packet Logs {#sec:packetlogsHowTo}
285 
286 
287 If things don't work as expected, having a look at the data exchanged
288 between BTstack and the Bluetooth chipset often helps.
289 
290 For this, BTstack provides a configurable packet logging mechanism via hci_dump.h:
291 
292  // formats: HCI_DUMP_BLUEZ, HCI_DUMP_PACKETLOGGER, HCI_DUMP_STDOUT
293  void hci_dump_open(const char *filename, hci_dump_format_t format);
294 
295 On POSIX systems, you can call *hci_dump_open* with a path and *HCI_DUMP_BLUEZ*
296 or *HCI_DUMP_PACKETLOGGER* in the setup, i.e., before entering the run loop.
297 The resulting file can be analyzed with Wireshark
298 or the Apple's PacketLogger tool.
299 
300 On embedded systems without a file system, you still can call *hci_dump_open(NULL, HCI_DUMP_STDOUT)*.
301 It will log all HCI packets to the consolve via printf.
302 If you capture the console output, incl. your own debug messages, you can use
303 the create_packet_log.py tool in the tools folder to convert a text output into a
304 PacketLogger file.
305 
306 In addition to the HCI packets, you can also enable BTstack's debug information by adding
307 
308  #define ENABLE_LOG_INFO
309  #define ENABLE_LOG_ERROR
310 
311 to the btstack-config.h and recompiling your application.
