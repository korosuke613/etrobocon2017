6 ## GAP - Generic Access Profile: Classic
7 
8 
9 The GAP profile defines how devices find each other and establish a
10 secure connection for other profiles. As mentioned before, the GAP
11 functionality is split between and . Please check both.

13 ### Become discoverable
14 
15 A remote unconnected Bluetooth device must be set as “discoverable” in
16 order to be seen by a device performing the inquiry scan. To become
17 discoverable, an application can call *hci_discoverable_control* with
18 input parameter 1. If you want to provide a helpful name for your
19 device, the application can set its local name by calling
20 $gap_set_local_name$. To save energy, you may set the device as
21 undiscoverable again, once a connection is established. See Listing
22 [below](#lst:Discoverable) for an example.
23 
24 ~~~~ {#lst:Discoverable .c caption="{Setting discoverable mode.}"}
25 
26  int main(void){
27  ...
28  // make discoverable
29  hci_discoverable_control(1);
30  run_loop_execute();
31  return 0;
32  }
33  void packet_handler (uint8_t packet_type, uint8_t *packet, uint16_t size){
34  ...
35  switch(state){
36  case W4_CHANNEL_COMPLETE:
37  // if connection is successful, make device undiscoverable
38  hci_discoverable_control(0);
39  ...
40  }
41  }
42 ~~~~

44 ### Discover remote devices {#sec:GAPdiscoverRemoteDevices}
45 
46 To scan for remote devices, the *hci_inquiry* command is used. Found
47 remote devices are reported as a part of:
48 
49 - HCI_EVENT_INQUIRY_RESULT,
50 
51 - HCI_EVENT-_INQUIRY_RESULT_WITH_RSSI, or
52 
53 - HCI_EVENT_EXTENDED_INQUIRY_RESPONSE events.
54 
55 Each response contains at least the Bluetooth address, the class of device, the page scan
56 repetition mode, and the clock offset of found device. The latter events
57 add information about the received signal strength or provide the
58 Extended Inquiry Result (EIR). A code snippet is shown in Listing
59 [below](#lst:DiscoverDevices).
60 
61 ~~~~ {#lst:DiscoverDevices .c caption="{Discover remote devices.}"}
62 
63  void print_inquiry_results(uint8_t *packet){
64  int event = packet[0];
65  int numResponses = packet[2];
66  uint16_t classOfDevice, clockOffset;
67  uint8_t rssi, pageScanRepetitionMode;
68  for (i=0; i<numResponses; i++){
69  bt_flip_addr(addr, &packet[3+i*6]);
70  pageScanRepetitionMode = packet [3 + numResponses*6 + i];
71  if (event == HCI_EVENT_INQUIRY_RESULT){
72  classOfDevice = READ_BT_24(packet, 3 + numResponses*(6+1+1+1) + i*3);
73  clockOffset = READ_BT_16(packet, 3 + numResponses*(6+1+1+1+3) + i*2) & 0x7fff;
74  rssi = 0;
75  } else {
76  classOfDevice = READ_BT_24(packet, 3 + numResponses*(6+1+1) + i*3);
77  clockOffset = READ_BT_16(packet, 3 + numResponses*(6+1+1+3) + i*2) & 0x7fff;
78  rssi = packet [3 + numResponses*(6+1+1+3+2) + i*1];
79  }
80  printf("Device found: %s with COD: 0x%06x, pageScan %u, clock offset 0x%04x, rssi 0x%02x\n", bd_addr_to_str(addr), classOfDevice, pageScanRepetitionMode, clockOffset, rssi);
81  }
82  }
83 
84  void packet_handler (uint8_t packet_type, uint8_t *packet, uint16_t size){
85  ...
86  switch (event) {
87  case HCI_STATE_WORKING:
88  hci_send_cmd(&hci_write_inquiry_mode, 0x01); // with RSSI
89  break;
90  case HCI_EVENT_COMMAND_COMPLETE:
91  if (COMMAND_COMPLETE_EVENT(packet, hci_write_inquiry_mode) ) {
92  start_scan();
93  }
94  case HCI_EVENT_COMMAND_STATUS:
95  if (COMMAND_STATUS_EVENT(packet, hci_write_inquiry_mode) ) {
96  printf("Ignoring error (0x%x) from hci_write_inquiry_mode.\n", packet[2]);
97  hci_send_cmd(&hci_inquiry, HCI_INQUIRY_LAP, INQUIRY_INTERVAL, 0);
98  }
99  break;
100  case HCI_EVENT_INQUIRY_RESULT:
101  case HCI_EVENT_INQUIRY_RESULT_WITH_RSSI:
102  print_inquiry_results(packet);
103  break;
104  ...
105  }
106  }
107 ~~~~
108 
109 By default, neither RSSI values nor EIR are reported. If the Bluetooth
110 device implements Bluetooth Specification 2.1 or higher, the
111 *hci_write_inquiry_mode* command enables reporting of this advanced
112 features (0 for standard results, 1 for RSSI, 2 for RSSI and EIR).
113 
114 A complete GAP inquiry example is provided [here](examples/generated/#sec:gapinquiryExample).

116 ### Pairing of Devices
117 
118 By default, Bluetooth communication is not authenticated, and any device
119 can talk to any other device. A Bluetooth device (for example, cellular
120 phone) may choose to require authentication to provide a particular
121 service (for example, a Dial-Up service). The process of establishing
122 authentication is called pairing. Bluetooth provides two mechanism for
123 this.
124 
125 On Bluetooth devices that conform to the Bluetooth v2.0 or older
126 specification, a PIN code (up to 16 bytes ASCII) has to be entered on
127 both sides. This isn’t optimal for embedded systems that do not have
128 full I/O capabilities. To support pairing with older devices using a
129 PIN, see Listing [below](#lst:PinCodeRequest).
130 
131 ~~~~ {#lst:PinCodeRequest .c caption="{PIN code request.}"}
132 
133  void packet_handler (uint8_t packet_type, uint8_t *packet, uint16_t size){
134  ...
135  switch (event) {
136  case HCI_EVENT_PIN_CODE_REQUEST:
137  // inform about pin code request
138  printf("Pin code request - using '0000'\n\r");
139  bt_flip_addr(bd_addr, &packet[2]);
140 
141  // baseband address, pin length, PIN: c-string
142  hci_send_cmd(&hci_pin_code_request_reply, &bd_addr, 4, "0000");
143  break;
144  ...
145  }
146  }
147 ~~~~
148 
149 The Bluetooth v2.1 specification introduces Secure Simple Pairing (SSP),
150 which is a better approach as it both improves security and is better
151 adapted to embedded systems. With SSP, the devices first exchange their
152 IO Capabilities and then settle on one of several ways to verify that
153 the pairing is legitimate. If the Bluetooth device supports SSP, BTstack
154 enables it by default and even automatically accepts SSP pairing
155 requests. Depending on the product in which BTstack is used, this may
156 not be desired and should be replaced with code to interact with the
157 user.
158 
159 Regardless of the authentication mechanism (PIN/SSP), on success, both
160 devices will generate a link key. The link key can be stored either in
161 the Bluetooth module itself or in a persistent storage, see
162 [here](porting/#sec:persistentStoragePorting). The next time the device connects and
163 requests an authenticated connection, both devices can use the
164 previously generated link key. Please note that the pairing must be
165 repeated if the link key is lost by one device.
166 
167 ### Dedicated Bonding
168 
169 Aside from the regular bonding, Bluetooth also provides the concept of
170 “dedicated bonding”, where a connection is established for the sole
171 purpose of bonding the device. After the bonding process is over, the
172 connection will be automatically terminated. BTstack supports dedicated
173 bonding via the *gap_dedicated_bonding* function.

175 ## SPP - Serial Port Profile
176 
177 
178 The SPP profile defines how to set up virtual serial ports and connect
179 two Bluetooth enabled devices.
180 
181 ### Accessing an SPP Server on a remote device
182 
183 To access a remote SPP server, you first need to query the remote device
184 for its SPP services. Section [on querying remote SDP service](#sec:querySDPProtocols)
185 shows how to query for all RFCOMM channels. For SPP, you can do the same
186 but use the SPP UUID 0x1101 for the query. After you have identified the
187 correct RFCOMM channel, you can create an RFCOMM connection as shown
188 [here](protocols/#sec:rfcommClientProtocols).
189 
190 ### Providing an SPP Server
191 
192 To provide an SPP Server, you need to provide an RFCOMM service with a
193 specific RFCOMM channel number as explained in section on
194 [RFCOMM service](protocols/#sec:rfcommServiceProtocols). Then, you need to create
195 an SDP record for it and publish it with the SDP server by calling
196 *sdp_register_service_internal*. BTstack provides the
197 *sdp_create_spp_service* function in that requires an empty buffer of
198 approximately 200 bytes, the service channel number, and a service name.
199 Have a look at the [SPP Counter example](examples/generated/#sec:sppcounterExample].

202 ## PAN - Personal Area Networking Profile {#sec:panProfiles}
203 
204 
205 The PAN profile uses BNEP to provide on-demand networking capabilities
206 between Bluetooth devices. The PAN profile defines the following roles:
207 
208 - PAN User (PANU)
209 
210 - Network Access Point (NAP)
211 
212 - Group Ad-hoc Network (GN)
213 
214 PANU is a Bluetooth device that communicates as a client with GN, or
215 NAP, or with another PANU Bluetooth device, through a point-to-point
216 connection. Either the PANU or the other Bluetooth device may terminate
217 the connection at anytime.
218 
219 NAP is a Bluetooth device that provides the service of routing network
220 packets between PANU by using BNEP and the IP routing mechanism. A NAP
221 can also act as a bridge between Bluetooth networks and other network
222 technologies by using the Ethernet packets.
223 
224 The GN role enables two or more PANUs to interact with each other
225 through a wireless network without using additional networking hardware.
226 The devices are connected in a piconet where the GN acts as a master and
227 communicates either point-to-point or a point-to-multipoint with a
228 maximum of seven PANU slaves by using BNEP.
229 
230 Currently, BTstack supports only PANU.
231 
232 ### Accessing a remote PANU service
233 
234 To access a remote PANU service, you first need perform an SDP query to
235 get the L2CAP PSM for the requested PANU UUID. With these two pieces of
236 information, you can connect BNEP to the remote PANU service with the
237 *bnep_connect* function. The Section on [PANU Demo example](#exaples/#sec:panudemoExample)
238 shows how this is accomplished.
239 
240 ### Providing a PANU service
241 
242 To provide a PANU service, you need to provide a BNEP service with the
243 service UUID, e.g. the PANU UUID, and a a maximal ethernet frame size,
244 as explained in Section [on BNEP service](protocols/#sec:bnepServiceProtocols). Then, you need to
245 create an SDP record for it and publish it with the SDP server by
246 calling *sdp_register_service_internal*. BTstack provides the
247 *pan_create_panu_service* function in *src/pan.c* that requires an
248 empty buffer of approximately 200 bytes, a description, and a security
249 description.

251 ## GAP LE - Generic Access Profile for Low Energy
252 
253 
254 As with GAP for Classic, the GAP LE profile defines how to discover and
255 how to connect to a Bluetooth Low Energy device. There are several GAP
256 roles that a Bluetooth device can take, but the most important ones are
257 the Central and the Peripheral role. Peripheral devices are those that
258 provide information or can be controlled. Central devices are those that
259 consume information or control the peripherals. Before the connection
260 can be established, devices are first going through an advertising
261 process.
262 
263 ### Private addresses.
264 
265 To better protect privacy, an LE device can choose to use a private i.e.
266 random Bluetooth address. This address changes at a user-specified rate.
267 To allow for later reconnection, the central and peripheral devices will
268 exchange their Identity Resolving Keys (IRKs) during bonding. The IRK is
269 used to verify if a new address belongs to a previously bonded device.
270 
271 To toggle privacy mode using private addresses, call the
272 *gap_random_address_set_mode* function. The update period can be set
273 with *gap_random_address_set_update_period*.
274 
275 After a connection is established, the Security Manager will try to
276 resolve the peer Bluetooth address as explained in Section on
277 [SMP](protocols/#sec:smpProtocols).
278 
279 ### Advertising and Discovery
280 
281 An LE device is discoverable and connectable, only if it periodically
282 sends out Advertisements. An advertisement contains up to 31 bytes of
283 data. To configure and enable advertisement broadcast, the following GAP
284 functions can be used:
285 
286 - *gap_advertisements_set_data*
287 
288 - *gap_advertisements_set_params*
289 
290 - *gap_advertisements_enable*
291 
292 Please have a look at the [SPP and LE
293 Counter example](examples/generated/#sec:sppandlecounterExample).
294 
295 In addition to the Advertisement data, a device in the peripheral role
296 can also provide Scan Response data, which has to be explicitly queried
297 by the central device. It can be provided with the
298 *hci_le_set_scan_response_data*.
299 
300 The scan parameters can be set with
301 *le_central_set_scan_parameters*. The scan can be started/stopped
302 with *le_central_start_scan*/*le_central_stop_scan*.
303 
304 Finally, if a suitable device is found, a connection can be initiated by
305 calling *le_central_connect*. In contrast to Bluetooth classic, there
306 is no timeout for an LE connection establishment. To cancel such an
307 attempt, *le_central_connect_cancel* has be be called.
308 
309 By default, a Bluetooth device stops sending Advertisements when it gets
310 into the Connected state. However, it does not start broadcasting
311 advertisements on disconnect again. To re-enable it, please send the
312 *hci_le_set_advertise_enable* again .

314 ## GATT - Generic Attribute Profile
315 
316 
317 The GATT profile uses ATT Attributes to represent a hierarchical
318 structure of GATT Services and GATT Characteristics. Each Service has
319 one or more Characteristics. Each Characteristic has meta data attached
320 like its type or its properties. This hierarchy of Characteristics and
321 Services are queried and modified via ATT operations.
322 
323 GATT defines both a server and a client role. A device can implement one
324 or both GATT roles.
325 
326 
327 ### GATT Client {#sec:GATTClientProfiles}
328 
329 The GATT Client is used to discover services, and their characteristics
330 and descriptors on a peer device. It can also subscribe for
331 notifications or indications that the characteristic on the GATT server
332 has changed its value.
333 
334 To perform GATT queries, provides a rich interface. Before calling
335 queries, the GATT client must be initialized with *gatt_client_init*
336 once.
337 
338 To allow for modular profile implementations, GATT client can be used
339 independently by multiple entities.
340 
341 To use it by a GATT client, you register a packet handler with
342 *gatt_client_register_packet_ handler*. The return value of that is
343 a GATT client ID which has to be provided in all queries.
344 
345 After an LE connection was created using the GAP LE API, you can query
346 for the connection MTU with *gatt_client_get_mtu*.
347 
348 GATT queries cannot be interleaved. Therefore, you can check if you can
349 perform a GATT query on a particular connection using
350 *gatt_client_is_ready*. As a result to a GATT query, zero to many
351 *le_event*s are returned before a *GATT_QUERY_COMPLETE* event
352 completes the query.
353 
354 For more details on the available GATT queries, please consult
355 [GATT Client API](#sec:gattClientAPIAppendix).
356

358 ### GATT Server {#sec:GATTServerProfiles}
359 
360 The GATT server stores data and accepts GATT client requests, commands
361 and confirmations. The GATT server sends responses to requests and when
362 configured, sends indication and notifications asynchronously to the
363 GATT client.
364 
365 To save on both code space and memory, BTstack does not provide a GATT
366 Server implementation. Instead, a textual description of the GATT
367 profile is directly converted into a compact internal ATT Attribute
368 database by a GATT profile compiler. The ATT protocol server -
369 implemented by and - answers incoming ATT requests based on information
370 provided in the compiled database and provides read- and write-callbacks
371 for dynamic attributes.
372 
373 GATT profiles are defined by a simple textual comma separated value
374 (.csv) representation. While the description is easy to read and edit,
375 it is compact and can be placed in ROM.
376 
377 The current format is shown in Listing [below](#lst:GATTServerProfile).
378 
379 ~~~~ {#lst:GATTServerProfile .c caption="{GATT profile.}"}
380 
381  PRIMARY_SERVICE, {SERVICE_UUID}
382  CHARACTERISTIC, {ATTRIBUTE_TYPE_UUID}, {PROPERTIES}, {VALUE}
383  CHARACTERISTIC, {ATTRIBUTE_TYPE_UUID}, {PROPERTIES}, {VALUE}
384  ...
385  PRIMARY_SERVICE, {SERVICE_UUID}
386  CHARACTERISTIC, {ATTRIBUTE_TYPE_UUID}, {PROPERTIES}, {VALUE}
387  ...
388 ~~~~
389 
390 Properties can be a list of READ $|$ WRITE $|$ WRITE_WITHOUT_RESPONSE
391 $|$ NOTIFY $|$ INDICATE $|$ DYNAMIC.
392 
393 Value can either be a string (“this is a string”), or, a sequence of hex
394 bytes (e.g. 01 02 03).
395 
396 UUIDs are either 16 bit (1800) or 128 bit
397 (00001234-0000-1000-8000-00805F9B34FB).
398 
399 Reads/writes to a Characteristic that is defined with the DYNAMIC flag,
400 are forwarded to the application via callback. Otherwise, the
401 Characteristics cannot be written and it will return the specified
402 constant value.
403 
404 Adding NOTIFY and/or INDICATE automatically creates an addition Client
405 Configuration Characteristic.
406 
407 To require encryption or authentication before a Characteristic can be
408 accessed, you can add ENCRYPTION_KEY_SIZE_X - with $X \in [7..16]$ -
409 or AUTHENTICATION_REQUIRED.
410 
411 BTstack only provides an ATT Server, while the GATT Server logic is
412 mainly provided by the GATT compiler. While GATT identifies
413 Characteristics by UUIDs, ATT uses Handles (16 bit values). To allow to
414 identify a Characteristic without hard-coding the attribute ID, the GATT
415 compiler creates a list of defines in the generated \*.h file.
