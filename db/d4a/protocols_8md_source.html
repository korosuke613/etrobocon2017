1 
2 BTstack is a modular dual-mode Bluetooth stack, supporting both
3 Bluetooth Basic Rate/Enhanced Date Rate (BR/EDR) as well as Bluetooth
4 Low Energy (LE). The BR/EDR technology, also known as Classic Bluetooth,
5 provides a robust wireless connection between devices designed for high
6 data rates. In contrast, the LE technology has a lower throughput but
7 also lower energy consumption, faster connection setup, and the ability
8 to connect to more devices in parallel.
9 
10 Whether Classic or LE, a Bluetooth device implements one or more
11 Bluetooth profiles. A Bluetooth profile specifies how one or more
12 Bluetooth protocols are used to achieve its goals. For example, every
13 Bluetooth device must implement the Generic Access Profile (GAP), which
14 defines how devices find each other and how they establish a connection.
15 This profile mainly make use of the Host Controller Interface (HCI)
16 protocol, the lowest protocol in the stack hierarchy which implements a
17 command interface to the Bluetooth chipset.
18 
19 In addition to GAP, a popular Classic Bluetooth example would be a
20 peripheral devices that can be connected via the Serial Port Profile
21 (SPP). SPP basically specifies that a compatible device should provide a
22 Service Discovery Protocol (SDP) record containing an RFCOMM channel
23 number, which will be used for the actual communication.
24 
25 Similarly, for every LE device, the Generic Attribute Profile (GATT)
26 profile must be implemented in addition to GAP. GATT is built on top of
27 the Attribute Protocol (ATT), and defines how one device can interact
28 with GATT Services on a remote device.
29 
30 So far, the most popular use of BTstack is in peripheral devices that
31 can be connected via SPP (Android 2.0 or higher) and GATT (Android 4.3
32 or higher, and iOS 5 or higher). If higher data rates are required
33 between a peripheral and iOS device, the iAP1 and iAP2 protocols of the
34 Made for iPhone program can be used instead of GATT. Please contact us
35 directly for information on BTstack and MFi.
36 
37 Figure {@fig:BTstackProtocolArchitecture} depicts Bluetooth protocols
38 and profiles that are currently implemented by BTstack.
39 In the following, we first explain how the various Bluetooth protocols
40 are used in BTstack. In the next chapter, we go over the profiles.

45 ## HCI - Host Controller Interface
46 
47 The HCI protocol provides a command interface to the Bluetooth chipset.
48 In BTstack, the HCI implementation also keeps track of all active
49 connections and handles the fragmentation and re-assembly of higher
50 layer (L2CAP) packets.
51 
52 Please note, that an application rarely has to send HCI commands on its
53 own. Instead, BTstack provides convenience functions in GAP and higher
54 level protocols use HCI automatically. E.g. to set the name, you can
55 call *gap_set_local_name()* before powering up. The main use of HCI
56 commands in application is during the startup phase to configure special
57 features that are not available via the GAP API yet.
58 
59 However, as many features of the GAP Classic can be achieved by sending
60 a single HCI command, not all GAP convenience functions are listed in .
61 If there’s no special GAP function, please consider sending the HCI
62 command directly, as explained in the following.
63 
64 ### Defining custom HCI command templates
65 
66 Each HCI command is assigned a 2-byte OpCode used to uniquely identify
67 different types of commands. The OpCode parameter is divided into two
68 fields, called the OpCode Group Field (OGF) and OpCode Command Field
69 (OCF), see [Bluetooth Specification](https://www.bluetooth.org/Technical/Specifications/adopted.htm) -
70 Core Version 4.0, Volume 2, Part E, Chapter 5.4.
71 
72 Listing [below](#lst:hciOGFs) shows the OGFs provided by BTstack in file [src/hci.h]():
73 
74 ~~~~ {#lst:hciOGFs .c caption="{HCI OGFs provided by BTstack.}"}
75 
76  #define OGF_LINK_CONTROL 0x01
77  #define OGF_LINK_POLICY 0x02
78  #define OGF_CONTROLLER_BASEBAND 0x03
79  #define OGF_INFORMATIONAL_PARAMETERS 0x04
80  #define OGF_LE_CONTROLLER 0x08
81  #define OGF_BTSTACK 0x3d
82  #define OGF_VENDOR 0x3f
83 ~~~~
84 
85 For all existing Bluetooth
86 commands and their OCFs see [Bluetooth Specification](https://www.bluetooth.org/Technical/Specifications/adopted.htm) -
87 Core Version 4.0, Volume 2, Part E, Chapter 7.
88 
89 In a HCI command packet, the OpCode is followed by parameter total
90 length, and the actual parameters. The OpCode of a command can be
91 calculated using the OPCODE macro. BTstack provides the *hci_cmd_t*
92 struct as a compact format to define HCI command packets, see
93 Listing [below](#lst:HCIcmdTemplate), and [include/btstack/hci_cmds.h]()
94 file in the source code.
95 
96 ~~~~ {#lst:HCIcmdTemplate .c caption="{HCI command struct.}"}
97 
98  // Calculate combined ogf/ocf value.
99  #define OPCODE(ogf, ocf) (ocf | ogf << 10)
100 
101  // Compact HCI Command packet description.
102  typedef struct {
103  uint16_t opcode;
104  const char *format;
105  } hci_cmd_t;
106 ~~~~
107 
108 Listing [below](#lst:HCIcmdExample) illustrates the *hci_write_local_name* HCI
109 command template from library:
110 
111 ~~~~ {#lst:HCIcmdExample .c caption="{HCI command example.}"}
112 
113  // Sets local Bluetooth name
114  const hci_cmd_t hci_write_local_name = {
115  OPCODE(OGF_CONTROLLER_BASEBAND, 0x13), "N"
116  // Local name (UTF-8, Null Terminated, max 248 octets)
117  };
118 ~~~~
119 
120 It uses OGF_CONTROLLER_BASEBAND as OGF,
121 0x13 as OCF, and has one parameter with format “N” indicating a null
122 terminated UTF-8 string. Table {@tbl:hciCmdParamSpecifier} lists the format
123 specifiers supported by BTstack. Check for other predefined HCI commands
124 and info on their parameters.
125 
126  ------------------- ----------------------------------------------------
127  Format Specifier Description
128  1,2,3,4 one to four byte value
129  A 31 bytes advertising data
130  B Bluetooth Baseband Address
131  D 8 byte data block
132  E Extended Inquiry Information 240 octets
133  H HCI connection handle
134  N Name up to 248 chars, UTF8 string, null terminated
135  P 16 byte Pairing code, e.g. PIN code or link key
136  S Service Record (Data Element Sequence)

142 ### Sending HCI command based on a template {#sec:sendingHCIProtocols}
143 
144 You can use the *hci_send_cmd* function to send HCI command based on a
145 template and a list of parameters. However, it is necessary to check
146 that the outgoing packet buffer is empty and that the Bluetooth module
147 is ready to receive the next command - most modern Bluetooth modules
148 only allow to send a single HCI command. This can be done by calling
149 *hci_can_send_command_packet_now()* function, which returns true,
150 if it is ok to send.
151 
152 Listing [below](#lst:HCIcmdExampleLocalName) illustrates how to manually set the
153 device name with the HCI Write Local Name command.
154 
155 ~~~~ {#lst:HCIcmdExampleLocalName .c caption="{Sending HCI command example.}"}
156 
157  if (hci_can_send_packet_now(HCI_COMMAND_DATA_PACKET)){
158  hci_send_cmd(&hci_write_local_name, "BTstack Demo");
159  }
160 ~~~~
161 
162 Please note, that an application rarely has to send HCI commands on its
163 own. Instead, BTstack provides convenience functions in GAP and higher
164 level protocols use HCI automatically.

167 ## L2CAP - Logical Link Control and Adaptation Protocol
168 
169 
170 The L2CAP protocol supports higher level protocol multiplexing and
171 packet fragmentation. It provides the base for the RFCOMM and BNEP
172 protocols. For all profiles that are officially supported by BTstack,
173 L2CAP does not need to be used directly. For testing or the development
174 of custom protocols, it’s helpful to be able to access and provide L2CAP
175 services however.
176 
177 ### Access an L2CAP service on a remote device
178 
179 L2CAP is based around the concept of channels. A channel is a logical
180 connection on top of a baseband connection. Each channel is bound to a
181 single protocol in a many-to-one fashion. Multiple channels can be bound
182 to the same protocol, but a channel cannot be bound to multiple
183 protocols. Multiple channels can share the same baseband connection.
184 
185 To communicate with an L2CAP service on a remote device, the application
186 on a local Bluetooth device initiates the L2CAP layer using the
187 *l2cap_init* function, and then creates an outgoing L2CAP channel to
188 the PSM of a remote device using the *l2cap_create_channel_internal*
189 function. The *l2cap_create_channel_internal* function will initiate
190 a new baseband connection if it does not already exist. The packet
191 handler that is given as an input parameter of the L2CAP create channel
192 function will be assigned to the new outgoing L2CAP channel. This
193 handler receives the L2CAP_EVENT_CHANNEL_OPENED and
194 L2CAP_EVENT_CHANNEL_CLOSED events and L2CAP data packets, as shown
195 in Listing [below](#lst:L2CAPremoteService).
196 
197 
198 ~~~~ {#lst:L2CAPremoteService .c caption="{Accessing an L2CAP service on a remote device.}"}
199 
200  btstack_packet_handler_t l2cap_packet_handler;
201 
202  void btstack_setup(){
203  ...
204  l2cap_init();
205  }
206 
207  void create_outgoing_l2cap_channel(bd_addr_t address, uint16_t psm, uint16_t mtu){
208  l2cap_create_channel_internal(NULL, l2cap_packet_handler, remote_bd_addr, psm, mtu);
209  }
210 
211  void l2cap_packet_handler(uint8_t packet_type, uint16_t channel, uint8_t *packet, uint16_t size){
212  if (packet_type == HCI_EVENT_PACKET &&
213  packet[0] == L2CAP_EVENT_CHANNEL_OPENED){
214  if (packet[2]) {
215  printf("Connection failed\n\r");
216  return;
217  }
218  printf("Connected\n\r");
219  }
220  if (packet_type == L2CAP_DATA_PACKET){
221  // handle L2CAP data packet
222  return;
223  }
224  }
225 ~~~~

227 ### Provide an L2CAP service
228 
229 To provide an L2CAP service, the application on a local Bluetooth device
230 must init the L2CAP layer and register the service with
231 *l2cap_register_service_internal*. From there on, it can wait for
232 incoming L2CAP connections. The application can accept or deny an
233 incoming connection by calling the *l2cap_accept_connection_internal*
234 and *l2cap_deny_connection_internal* functions respectively. If a
235 connection is accepted and the incoming L2CAP channel gets successfully
236 opened, the L2CAP service can send L2CAP data packets to the connected
237 device with *l2cap_send_internal*.
238 
239 Sending of L2CAP data packets may fail due to a full internal BTstack
240 outgoing packet buffer, or if the ACL buffers in the Bluetooth module
241 become full, i.e., if the application is sending faster than the packets
242 can be transferred over the air. In such case, the application can try
243 sending again upon reception of DAEMON_EVENT_HCI_PACKET_SENT or
244 L2CAP_EVENT_CREDITS event. The first event signals that the internal
245 BTstack outgoing buffer became free again, the second one signals the
246 same for ACL buffers in the Bluetooth chipset. Listing [below](#lst:L2CAPService)
247 provides L2CAP service example code.
248 
249 
250 ~~~~ {#lst:L2CAPService .c caption="{Providing an L2CAP service.}"}
251 
252  void btstack_setup(){
253  ...
254  l2cap_init();
255  l2cap_register_service_internal(NULL, packet_handler, 0x11,100);
256  }
257 
258  void packet_handler (uint8_t packet_type, uint16_t channel, uint8_t *packet, uint16_t size){
259  ...
260  if (packet_type == L2CAP_DATA_PACKET){
261  // handle L2CAP data packet
262  return;
263  }
264  switch(event){
265  ...
266  case L2CAP_EVENT_INCOMING_CONNECTION:
267  bt_flip_addr(event_addr, &packet[2]);
268  handle = READ_BT_16(packet, 8);
269  psm = READ_BT_16(packet, 10);
270  local_cid = READ_BT_16(packet, 12);
271  printf("L2CAP incoming connection requested.");
272  l2cap_accept_connection_internal(local_cid);
273  break;
274  case L2CAP_EVENT_CHANNEL_OPENED:
275  bt_flip_addr(event_addr, &packet[3]);
276  psm = READ_BT_16(packet, 11);
277  local_cid = READ_BT_16(packet, 13);
278  handle = READ_BT_16(packet, 9);
279  if (packet[2] == 0) {
280  printf("Channel successfully opened.");
281  } else {
282  printf("L2CAP connection failed. status code.");
283  }
284  break;
285  case L2CAP_EVENT_CREDITS:
286  case DAEMON_EVENT_HCI_PACKET_SENT:
287  tryToSend();
288  break;
289  case L2CAP_EVENT_CHANNEL_CLOSED:
290  break;
291  }
292  }
293 ~~~~

303 ## RFCOMM - Radio Frequency Communication Protocol
304 
305 The Radio frequency communication (RFCOMM) protocol provides emulation
306 of serial ports over the L2CAP protocol. and reassembly. It is the base
307 for the Serial Port Profile and other profiles used for
308 telecommunication like Head-Set Profile, Hands-Free Profile, Object
309 Exchange (OBEX) etc.
310 
311 
312 ### RFCOMM flow control {#sec:flowControlProtocols}
313 
314 RFCOMM has a mandatory credit-based flow-control. This means that two
315 devices that established RFCOMM connection, use credits to keep track of
316 how many more RFCOMM data packets can be sent to each. If a device has
317 no (outgoing) credits left, it cannot send another RFCOMM packet, the
318 transmission must be paused. During the connection establishment,
319 initial credits are provided. BTstack tracks the number of credits in
320 both directions. If no outgoing credits are available, the RFCOMM send
321 function will return an error, and you can try later. For incoming data,
322 BTstack provides channels and services with and without automatic credit
323 management via different functions to create/register them respectively.
324 If the management of credits is automatic, the new credits are provided
325 when needed relying on ACL flow control - this is only useful if there
326 is not much data transmitted and/or only one physical connection is
327 used. If the management of credits is manual, credits are provided by
328 the application such that it can manage its receive buffers explicitly.
329 
330 
331 ### Access an RFCOMM service on a remote device {#sec:rfcommClientProtocols}
332 
333 To communicate with an RFCOMM service on a remote device, the
334 application on a local Bluetooth device initiates the RFCOMM layer using
335 the *rfcomm_init* function, and then creates an outgoing RFCOMM channel
336 to a given server channel on a remote device using the
337 *rfcomm_create_channel_internal* function. The
338 *rfcomm_create_channel_internal* function will initiate a new L2CAP
339 connection for the RFCOMM multiplexer, if it does not already exist. The
340 channel will automatically provide enough credits to the remote side. To
341 provide credits manually, you have to create the RFCOMM connection by
342 calling *rfcomm_create_channel_with_initial_credits_internal* -
343 see Section [on manual credit assignement](#sec:manualCreditsProtocols).
344 
345 The packet handler that is given as an input parameter of the RFCOMM
346 create channel function will be assigned to the new outgoing channel.
347 This handler receives the RFCOMM_EVENT_OPEN_CHANNEL_COMPLETE and
348 RFCOMM_EVENT_CHANNEL_CLOSED events, and RFCOMM data packets, as shown in
349 Listing [below](#lst:RFCOMMremoteService).
350 
351 
352 ~~~~ {#lst:RFCOMMremoteService .c caption="{RFCOMM handler for outgoing RFCOMM channel.}"}

358  }

362  }
363

365  if (packet_type == HCI_EVENT_PACKET && packet[0] == RFCOMM_EVENT_OPEN_CHANNEL_COMPLETE){
366  if (packet[2]) {

380 ### Provide an RFCOMM service {#sec:rfcommServiceProtocols}
381 
382 To provide an RFCOMM service, the application on a local Bluetooth
383 device must first init the L2CAP and RFCOMM layers and then register the
384 service with *rfcomm_register_service_internal*. From there on, it
385 can wait for incoming RFCOMM connections. The application can accept or
386 deny an incoming connection by calling the
387 *rfcomm_accept_connection_internal* and
388 *rfcomm_deny_connection_internal* functions respectively. If a
389 connection is accepted and the incoming RFCOMM channel gets successfully
390 opened, the RFCOMM service can send RFCOMM data packets to the connected
391 device with *rfcomm_send_internal* and receive data packets by the
392 packet handler provided by the *rfcomm_register_service_internal*
393 call.
394 
395 Sending of RFCOMM data packets may fail due to a full internal BTstack
396 outgoing packet buffer, or if the ACL buffers in the Bluetooth module
397 become full, i.e., if the application is sending faster than the packets
398 can be transferred over the air. In such case, the application can try
399 sending again upon reception of DAEMON_EVENT_HCI_PACKET_SENT or
400 RFCOMM_EVENT_CREDITS event. The first event signals that the internal
401 BTstack outgoing buffer became free again, the second one signals that
402 the remote side allowed to send another packet. Listing [below](#lst:RFCOMMService)
403 provides the RFCOMM service example code.
404 
405 
406 ~~~~ {#lst:RFCOMMService .c caption="{Providing an RFCOMM service.}"}
407 
408  void btstack_setup(){
409  ...
410  rfcomm_init();
411  rfcomm_register_service_internal(NULL, rfcomm_channel_nr, mtu);
412  }
413 
414  void packet_handler(uint8_t packet_type, uint8_t *packet, uint16_t size){
415  if (packet_type == RFCOMM_DATA_PACKET){
416  // handle RFCOMM data packets
417  return;
418  }
419  ...
420  switch (event) {
421  ...
422  case RFCOMM_EVENT_INCOMING_CONNECTION:
423  //data: event(8), len(8), address(48), channel(8), rfcomm_cid(16)
424  bt_flip_addr(event_addr, &packet[2]);
425  rfcomm_channel_nr = packet[8];
426  rfcomm_channel_id = READ_BT_16(packet, 9);
427  rfcomm_accept_connection_internal(rfcomm_channel_id);
428  break;
429  case RFCOMM_EVENT_OPEN_CHANNEL_COMPLETE:
430  // data: event(8), len(8), status (8), address (48), handle(16), server channel(8), rfcomm_cid(16), max frame size(16)
431  if (packet[2]) {
432  printf("RFCOMM channel open failed.");
433  break;
434  }
435  // data: event(8), len(8), status (8), address (48), handle (16), server channel(8), rfcomm_cid(16), max frame size(16)
436  rfcomm_channel_id = READ_BT_16(packet, 12);
437  mtu = READ_BT_16(packet, 14);
438  printf("RFCOMM channel open succeeded.");
439  break;
440  case RFCOMM_EVENT_CREDITS:
441  case DAEMON_EVENT_HCI_PACKET_SENT:
442  tryToSend();
443  break;
444 
445  case RFCOMM_EVENT_CHANNEL_CLOSED:
446  printf("Channel closed.");
447  rfcomm_channel_id = 0;
448  break;
449  }
450  }
451 ~~~~

453 ### Living with a single output buffer {#sec:singleBufferProtocols}
454 
455 Outgoing packets, both commands and data, are not queued in BTstack.
456 This section explains the consequences of this design decision for
457 sending data and why it is not as bad as it sounds.
458 
459 Independent from the number of output buffers, packet generation has to
460 be adapted to the remote receiver and/or maximal link speed. Therefore,
461 a packet can only be generated when it can get sent. With this
462 assumption, the single output buffer design does not impose additional
463 restrictions. In the following, we show how this is used for adapting
464 the RFCOMM send rate.
465 
466 Before sending data packets, check if RFCOMM can send them by calling rfcomm_can_send_packet_now, as shown in Listing [below](#lst:SingleOutputBufferTryToSend). L2CAP, BNEP, and ATT API offer similar functions.
467 
468 ~~~~ {#lst:SingleOutputBufferTryToSend .c caption="{Preparing and sending data.}"}
469  void prepareData(void){
470  ...
471  }
472 
473  void tryToSend(void){
474  if (!dataLen) return;
475  if (!rfcomm_channel_id) return;
476  if (!rfcomm_can_send_packet_now(rfcomm_channel_id)) return;
477 
478  int err = rfcomm_send_internal(rfcomm_channel_id, dataBuffer, dataLen);
479  if (err) {
480  log_error("rfcomm_send_internal -> error 0X%02x", err);
481  return;
482  }
483  // packet is sent prepare next one
484  prepareData();
485  }
486 ~~~~
487 
488 RFCOMM’s mandatory credit-based flow-control imposes an additional
489 constraint on sending a data packet - at least one new RFCOMM credit
490 must be available. BTstack signals the availability of a credit by
491 sending an RFCOMM credit (RFCOMM_EVENT_CREDITS) event.
492 
493 These two events represent two orthogonal mechanisms that deal with flow
494 control. Taking these mechanisms in account, the application should try
495 to send data packets when one of these two events is received. For an
496 RFCOMM example see Listing [below](#lst:SingleOutputBufferTryPH).
497 
498 
499 ~~~~ {#lst:SingleOutputBufferTryPH .c caption="{Resending data packets.}"}
500 
501  void packet_handler (uint8_t packet_type, uint16_t channel, uint8_t *packet, uint16_t size){
502  ...
503  switch(event){
504  case RFCOMM_EVENT_OPEN_CHANNEL_COMPLETE:
505  if (status) {
506  printf("RFCOMM channel open failed.");
507  } else {
508  rfcomm_channel_id = READ_BT_16(packet, 12);
509  rfcomm_mtu = READ_BT_16(packet, 14);
510  printf("RFCOMM channel opened, mtu = %u.", rfcomm_mtu);
511  }
512  break;
513  case RFCOMM_EVENT_CREDITS:
514  case DAEMON_EVENT_HCI_PACKET_SENT:
515  tryToSend();
516  break;
517  case RFCOMM_EVENT_CHANNEL_CLOSED:
518  rfcomm_channel_id = 0;
519  break;
520  ...
521  }
522  }
523 ~~~~
524 
525 If the management of credits is manual, credits are provided by the
526 application such that it can manage its receive buffers explicitly, see
527 Listing [below](#lst:explicitFlowControl).
528 
529 
530 ~~~~ {#lst:explicitFlowControl .c caption="{RFCOMM service with manual credit management.}"}
531  void btstack_setup(void){
532  ...
533  // init RFCOMM
534  rfcomm_init();
535  rfcomm_register_packet_handler(packet_handler);
536  // reserved channel, mtu=100, 1 credit
537  rfcomm_register_service_with_initial_credits_internal(NULL, rfcomm_channel_nr, 100, 1);
538  }
539 ~~~~
540 
541 Manual credit management is recommended when received RFCOMM data cannot
542 be processed immediately. In the [SPP flow control example](examples/generated/#sec:sppflowcontrolExample),
543 delayed processing of received data is
544 simulated with the help of a periodic timer. To provide new credits, you
545 call the *rfcomm_grant_credits* function with the RFCOMM channel ID
546 and the number of credits as shown in Listing [below](#lst:NewCredits).
547 
548 ~~~~ {#lst:NewCredits .c caption="{Granting RFCOMM credits.}"}
549  void processing(){
550  // process incoming data packet
551  ...
552  // provide new credit
553  rfcomm_grant_credits(rfcomm_channel_id, 1);
554  }
555 ~~~~
556 
557 Please note that providing single credits effectively reduces the credit-based
558 (sliding window) flow control to a stop-and-wait flow-control that
559 limits the data throughput substantially. On the plus side, it allows
560 for a minimal memory footprint. If possible, multiple RFCOMM buffers
561 should be used to avoid pauses while the sender has to wait for a new
562 credit.

565 ### Slowing down RFCOMM data reception {#sec:manualCreditsProtocols}
566 
567 RFCOMM’s credit-based flow-control can be used to adapt, i.e., slow down
568 the RFCOMM data to your processing speed. For incoming data, BTstack
569 provides channels and services with and without automatic credit
570 management. If the management of credits is automatic, new credits
571 are provided when needed relying on ACL flow control. This is only
572 useful if there is not much data transmitted and/or only one physical
573 connection is used. See Listing [below](#lst:automaticFlowControl).
574 
575 ~~~~ {#lst:automaticFlowControl .c caption="{RFCOMM service with automatic credit management.}"}
576  void btstack_setup(void){
577  ...
578  // init RFCOMM
579  rfcomm_init();
580  rfcomm_register_packet_handler(packet_handler);
581  rfcomm_register_service_internal(NULL, rfcomm_channel_nr, 100);

585 ## SDP - Service Discovery Protocol
586 
587 The SDP protocol allows to announce services and discover services
588 provided by a remote Bluetooth device.
589 
590 ### Create and announce SDP records
591 
592 BTstack contains a complete SDP server and allows to register SDP
593 records. An SDP record is a list of SDP Attribute *{ID, Value}* pairs
594 that are stored in a Data Element Sequence (DES). The Attribute ID is a
595 16-bit number, the value can be of other simple types like integers or
596 strings or can itselff contain other DES.
597 
598 To create an SDP record for an SPP service, you can call
599 *sdp_create_spp_service* from with a pointer to a buffer to store the
600 record, the server channel number, and a record name.
601 
602 For other types of records, you can use the other functions in , using
603 the data element *de_* functions. Listing [sdpCreate] shows how an SDP
604 record containing two SDP attributes can be created. First, a DES is
605 created and then the Service Record Handle and Service Class ID List
606 attributes are added to it. The Service Record Handle attribute is added
607 by calling the *de_add_number* function twice: the first time to add
608 0x0000 as attribute ID, and the second time to add the actual record
609 handle (here 0x1000) as attribute value. The Service Class ID List
610 attribute has ID 0x0001, and it requires a list of UUIDs as attribute
611 value. To create the list, *de_push_sequence* is called, which “opens”
612 a sub-DES. The returned pointer is used to add elements to this sub-DES.
613 After adding all UUIDs, the sub-DES is “closed” with
614 *de_pop_sequence*.
615 
616 
617 ### Query remote SDP service {#sec:querySDPProtocols}
618 
619 BTstack provides an SDP client to query SDP services of a remote device.
620 The SDP Client API is shown in [here](appendix/apis/#sec:sdpAPIAppendix). The
621 *sdp_client_query* function initiates an L2CAP connection to the
622 remote SDP server. Upon connect, a *Service Search Attribute* request
623 with a *Service Search Pattern* and a *Attribute ID List* is sent. The
624 result of the *Service Search Attribute* query contains a list of
625 *Service Records*, and each of them contains the requested attributes.
626 These records are handled by the SDP parser. The parser delivers
627 SDP_PARSER_ATTRIBUTE_VALUE and SDP_PARSER_COMPLETE events via a
628 registered callback. The SDP_PARSER_ATTRIBUTE_VALUE event delivers
629 the attribute value byte by byte.
630 
631 On top of this, you can implement specific SDP queries. For example,
632 BTstack provides a query for RFCOMM service name and channel number.
633 This information is needed, e.g., if you want to connect to a remote SPP
634 service. The query delivers all matching RFCOMM services, including its
635 name and the channel number, as well as a query complete event via a
636 registered callback, as shown in Listing [below](#lst:SDPClientRFCOMM).
637 
638 
639 ~~~~ {#lst:SDPClientRFCOMM .c caption="{Searching RFCOMM services on a remote device.}"}

643  void packet_handler (void * connection, uint8_t packet_type, uint16_t channel, uint8_t *packet, uint16_t size){

646  uint8_t event = packet[0];
647  switch (event) {

652  }
653  break;
654  default:
655  break;
656  }
657  }
658 
659  static void btstack_setup(){
660  ...
661  // init L2CAP
662  l2cap_init();
663  l2cap_register_packet_handler(packet_handler);
664  }

667  sdp_query_rfcomm_service_event_t * ve;

672  printf("Service name: '%s', RFCOMM port %u\n", ve->service_name, ve->channel_nr);

676  printf("Client query response done with status %d. \n", ce->status);

697 ## BNEP - Bluetooth Network Encapsulation Protocol
698 
699 The BNEP protocol is used to transport control and data packets over
700 standard network protocols such as TCP, IPv4 or IPv6. It is built on top
701 of L2CAP, and it specifies a minimum L2CAP MTU of 1691 bytes.
702 
703 ### Receive BNEP events
704 
705 To receive BNEP events, please register a packet handler with
706 *bnep_register_packet_handler*.
707 
708 
709 ### Access a BNEP service on a remote device {#sec:bnepClientProtocols}
710 
711 To connect to a remote BNEP service, you need to know its UUID. The set
712 of available UUIDs can be queried by a SDP query for the PAN profile.
713 Please see section on [PAN profile](profiles/#sec:panProfiles) for details.
714 With the remote UUID, you can create a connection using the *bnep_connect*
715 function. You’ll receive a *BNEP_EVENT_OPEN_CHANNEL_COMPLETE* on success or
716 failure.
717 
718 After the connection was opened successfully, you can send and receive
719 Ethernet packets. Before sending an Ethernet frame with *bnep_send*,
720 *bnep_can_send_packet_now* needs to return true. Ethernet frames
721 are received via the registered packet handler with packet type
722 *BNEP_DATA_PACKET*.
723 
724 BTstack BNEP implementation supports both network protocol filter and
725 multicast filters with *bnep_set_net_type_filter* and
726 *bnep_set_multicast_filter* respectively.
727 
728 Finally, to close a BNEP connection, you can call *bnep_disconnect*.
729 
730 
731 ### Provide BNEP service {#sec:bnepServiceProtocols}
732 
733 To provide a BNEP service, call *bnep_register_service* with the
734 provided service UUID and a max frame size.
735 
736 A *BNEP_EVENT_INCOMING_CONNECTION* event will mark that an incoming
737 connection is established. At this point you can start sending and
738 receiving Ethernet packets as described in the previous section.
739 
740 ## ATT - Attribute Protocol
741 
742 The ATT protocol is used by an ATT client to read and write attribute
743 values stored on an ATT server. In addition, the ATT server can notify
744 the client about attribute value changes. An attribute has a handle, a
745 type, and a set of properties, see Section [section:GATTServer].
746 
747 The Generic Attribute (GATT) profile is built upon ATT and provides
748 higher level organization of the ATT attributes into GATT Services and
749 GATT Characteristics. In BTstack, the complete ATT client functionality
750 is included within the GATT Client. On the server side, one ore more
751 GATT profiles are converted ahead of time into the corresponding ATT
752 attribute database and provided by the *att_server* implementation. The
753 constant data are automatically served by the ATT server upon client
754 request. To receive the dynamic data, such is characteristic value, the
755 application needs to register read and/or write callback. In addition,
756 notifications and indications can be sent. Please see Section on
757 [GATT client](profiles/#sec:GATTClientProfiles) for more.

760 ## SMP - Security Manager Protocol {#sec:smpProtocols}
761 
762 The SMP protocol allows to setup authenticated and encrypted LE
763 connection. After initialization and configuration, SMP handles security
764 related functions on it’s own but emits events when feedback from the
765 main app or the user is required. The two main tasks of the SMP protocol
766 are: bonding and identity resolving.
767 
768 ### Initialization
769 
770 To activate the security manager, call *sm_init()*.
771 
772 If you’re creating a product, you should also call *sm_set_ir()* and
773 *sm_set_er()* with a fixed random 16 byte number to create the IR and
774 ER key seeds. If possible use a unique random number per device instead
775 of deriving it from the product serial number or something similar. The
776 encryption key generated by the BLE peripheral will be ultimately
777 derived from the ER key seed. See
778 [Bluetooth Specification](https://www.bluetooth.org/Technical/Specifications/adopted.htm) -
779 Bluetooth Core V4.0, Vol 3, Part G, 5.2.2 for more details on deriving
780 the different keys. The IR key is used to identify a device if private,
781 resolvable Bluetooth addresses are used.
782 
783 ### Configuration
784 
785 To receive events from the Security Manager, a callback is necessary.
786 How to register this packet handler depends on your application
787 configuration.
788 
789 When *att_server* is used to provide a GATT/ATT service, *att_server*
790 registers itself as the Security Manager packet handler. Security
791 Manager events are then received by the application via the
792 *att_server* packet handler.
793 
794 If *att_server* is not used, you can directly register your packet
795 handler with the security manager by calling
796 *sm_register_packet_handler*.
797 
798 The default SMP configuration in BTstack is to be as open as possible:
799 
800 - accept all Short Term Key (STK) Generation methods,
801 
802 - accept encryption key size from 7..16 bytes,
803 
804 - expect no authentication requirements, and
805 
806 - IO Capabilities set to *IO_CAPABILITY_NO_INPUT_NO_OUTPUT*.
807 
808 You can configure these items by calling following functions
809 respectively:
810 
811 - *sm_set_accepted_stk_generation_methods*
812 
813 - *sm_set_encryption_key_size_range*
814 
815 - *sm_set_authentication_requirements*
816 
817 - *sm_set_io_capabilities*
818 
819 ### Identity Resolving
820 
821 Identity resolving is the process of matching a private, resolvable
822 Bluetooth address to a previously paired device using its Identity
823 Resolving (IR) key. After an LE connection gets established, BTstack
824 automatically tries to resolve the address of this device. During this
825 lookup, BTstack will emit the following events:
826 
827 - *SM_IDENTITY_RESOLVING_STARTED* to mark the start of a lookup,
828 
829 and later:
830 
831 - *SM_IDENTITY_RESOLVING_SUCCEEDED* on lookup success, or
832 
833 - *SM_IDENTITY_RESOLVING_FAILED* on lookup failure.
834 
835 ### Bonding process
836 
837 In Bluetooth LE, there are three main methods of establishing an
838 encrypted connection. From the most to the least secure, these are:
839 Out-of-Band (OOB) Data , Passkey, and Just Works.
840 
841 With OOB data, there needs to be a pre-shared secret 16 byte key. In
842 most cases, this is not an option, especially since popular OS like iOS
843 don’t provide a way to specify it. It some applications, where both
844 sides of a Bluetooth link are developed together, this could provide a
845 viable option.
846 
847 To provide OOB data, you can register an OOB data callback with
848 *sm_register_oob_data_callback*.
849 
850 Depending on the authentication requirements, available OOB data, and
851 the enabled STK generation methods, BTstack will request feedback from
852 the app in the form of an event:
853 
854 - *SM_PASSKEY_INPUT_NUMBER*: request user to input a passkey
855 
856 - *SM_PASSKEY_DISPLAY_NUMBER*: show a passkey to the user
857 
858 - *SM_JUST_WORKS_REQUEST*: request a user to accept a Just Works
859  pairing
860 
861 To stop the bonding process, *sm_bonding_decline* should be called.
862 Otherwise, *sm_just_works_confirm* or *sm_passkey_input* can be
863 called.
864 
865 After the bonding process, *SM_PASSKEY_DISPLAY_CANCEL* is emitted to
866 update the user interface.
