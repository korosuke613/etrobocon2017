2 While the run loop provided by BTstack is sufficient for new designs,
3 BTstack is often used with or added to existing projects. In this case,
4 the run loop, data sources, and timers may need to be adapted. The
5 following two sections provides a guideline for single and
6 multi-threaded environments.
7 
8 To simplify the discussion, we’ll consider an application split into
9 “Main ”, “Communication Logic”, and “BTstack”. The Communication Logic
10 contains the packet handler (PH) that handles all asynchronous events
11 and data packets from BTstack. The Main Application makes use of the
12 Communication Logic for its Bluetooth communication.
13 
14 
15 ## Adapting BTstack for Single-Threaded Environments {#sec:singlethreadingIntegration}
16 
17 
18 In a single-threaded environment, all application components run on the
19 same (single) thread and use direct function calls as shown in
20 Figure {@fig:BTstackSingle}.
21 
22 ![BTstack in single-threaded environment.](picts/singlethreading-btstack.png) {#fig:BTstackSingle}
23 
24 BTstack provides a basic run loop that supports the concept of data
25 sources and timers, which can be registered centrally. This works well
26 when working with a small MCU and without an operating system. To adapt
27 to a basic operating system or a different scheduler, BTstack’s run loop
28 can be implemented based on the functions and mechanism of the existing
29 system.
30 
31 Currently, we have two examples for this:
32 
33 - *run_loop_cocoa.c* is an implementation for the CoreFoundation
34  Framework used in OS X and iOS. All run loop functions are
35  implemented in terms of CoreFoundation calls, data sources and
36  timers are modeled as CFSockets and CFRunLoopTimer respectively.
37 
38 - *run_loop_posix.c* is an implementation for POSIX compliant
39  systems. The data sources are modeled as file descriptors and
40  managed in a linked list. Then, the*select* function is used to wait
41  for the next file descriptor to become ready or timer to expire.

44 ## Adapting BTstack for Multi-Threaded Environments {#sec:multithreadingIntegration}
45 
46 
47 The basic execution model of BTstack is a general while loop. Aside from
48 interrupt-driven UART and timers, everything happens in sequence. When
49 using BTstack in a multi-threaded environment, this assumption has to
50 stay valid - at least with respect to BTstack. For this, there are two
51 common options:
52 
53 
54 - The Communication Logic is implemented on a dedicated BTstack
55  thread, and the Main Application communicates with the BTstack
56  thread via application-specific messages over an Interprocess
57  Communication (IPC) as depicted in Figure {@fig:MTMonolithic}.
58  This option results in less code and quick adaption.
59 
60  ![BTstack in multi-threaded environment - monolithic solution.](picts/multithreading-monolithic.png) {#fig:MTMonolithic}
61 
62 - BTstack must be extended to run standalone, i.e, as a Daemon, on a
63  dedicated thread and the Main Application controls this daemon via
64  BTstack extended HCI command over IPC - this is used for the
65  non-embedded version of BTstack e.g., on the iPhone and it is depicted
66  in Figure {@fig:MTDaemon}. This option requires more code but provides
67  more flexibility.
68 
69  ![BTstack in multi-threaded environment - solution with daemon.](picts/multithreading-btdaemon.png) {#fig:MTDaemon}
