5 ## Time Abstraction Layer {#sec:timeAbstractionPorting}
6 
7 BTstack requires a way to learn about passing time.
8 *run_loop_embedded.c* supports two different modes: system ticks or a
9 system clock with millisecond resolution. BTstack’s timing requirements
10 are quite low as only Bluetooth timeouts in the second range need to be
11 handled.
12 
13 
14 ### Tick Hardware Abstraction {#sec:tickAbstractionPorting}
15 
16 
17 If your platform doesn’t require a system clock or if you already have a
18 system tick (as it is the default with CMSIS on ARM Cortex devices), you
19 can use that to implement BTstack’s time abstraction in
20 *include/btstack/hal_tick.h>*.
21 
22 For this, you need to define *HAVE_TICK* in *btstack-config.h*:
23 
24  #define HAVE_TICK
25 
26 Then, you need to implement the functions *hal_tick_init* and
27 *hal_tick_set_handler*, which will be called during the
28 initialization of the run loop.
29 
30  void hal_tick_init(void);
31  void hal_tick_set_handler(void (*tick_handler)(void));
32  int hal_tick_get_tick_period_in_ms(void);
33 
34 After BTstack calls *hal_tick_init()* and
35 *hal_tick_set_handler(tick_handler)*, it expects that the
36 *tick_handler* gets called every
37 *hal_tick_get_tick_period_in_ms()* ms.
38 
39 
40 ### Time MS Hardware Abstraction {#sec:timeMSAbstractionPorting}
41 
42 
43 If your platform already has a system clock or it is more convenient to
44 provide such a clock, you can use the Time MS Hardware Abstraction in
45 *include/btstack/hal_time_ms.h*.
46 
47 For this, you need to define *HAVE_TIME_MS* in *btstack-config.h*:
48 
49  #define HAVE_TIME_MS
50 
51 Then, you need to implement the function *hal_time_ms()*, which will
52 be called from BTstack’s run loop and when setting a timer for the
53 future. It has to return the time in milliseconds.
54 
55  uint32_t hal_time_ms(void);

58 ## Bluetooth Hardware Control API {#sec:btHWControlPorting}
59 
60 
61 The Bluetooth hardware control API can provide the HCI layer with a
62 custom initialization script, a vendor-specific baud rate change
63 command, and system power notifications. It is also used to control the
64 power mode of the Bluetooth module, i.e., turning it on/off and putting
65 to sleep. In addition, it provides an error handler *hw_error* that is
66 called when a Hardware Error is reported by the Bluetooth module. The
67 callback allows for persistent logging or signaling of this failure.
68 
69 Overall, the struct *bt_control_t* encapsulates common functionality
70 that is not covered by the Bluetooth specification. As an example, the
71 *bt_control_cc256x_in-stance* function returns a pointer to a control
72 struct suitable for the CC256x chipset.

76 ## HCI Transport Implementation {#sec:hciTransportPorting}
77 
78 
79 On embedded systems, a Bluetooth module can be connected via USB or an
80 UART port. BTstack implements two UART based protocols for carrying HCI
81 commands, events and data between a host and a Bluetooth module: HCI
82 UART Transport Layer (H4) and H4 with eHCILL support, a lightweight
83 low-power variant by Texas Instruments.
84 
85 
86 ### HCI UART Transport Layer (H4) {#sec:hciUARTPorting}
87 
88 
89 Most embedded UART interfaces operate on the byte level and generate a
90 processor interrupt when a byte was received. In the interrupt handler,
91 common UART drivers then place the received data in a ring buffer and
92 set a flag for further processing or notify the higher-level code, i.e.,
93 in our case the Bluetooth stack.
94 
95 Bluetooth communication is packet-based and a single packet may contain
96 up to 1021 bytes. Calling a data received handler of the Bluetooth stack
97 for every byte creates an unnecessary overhead. To avoid that, a
98 Bluetooth packet can be read as multiple blocks where the amount of
99 bytes to read is known in advance. Even better would be the use of
100 on-chip DMA modules for these block reads, if available.
101 
102 The BTstack UART Hardware Abstraction Layer API reflects this design
103 approach and the underlying UART driver has to implement the following
104 API:
105 
106  void hal_uart_dma_init(void);
107  void hal_uart_dma_set_block_received(void (*block_handler)(void));
108  void hal_uart_dma_set_block_sent(void (*block_handler)(void));
109  int hal_uart_dma_set_baud(uint32_t baud);
110  void hal_uart_dma_send_block(const uint8_t *buffer, uint16_t len);
111  void hal_uart_dma_receive_block(uint8_t *buffer, uint16_t len);
112 
113 
114 The main HCI H4 implementations for embedded system is
115 *hci_h4_transport-_dma* function. This function calls the following
116 sequence: *hal_uart_dma_init*, *hal_uart_dma_set_block_received*
117 and *hal_uart_dma_set_block_sent* functions. this sequence, the HCI
118 layer will start packet processing by calling
119 *hal_uart-_dma_receive_block* function. The HAL implementation is
120 responsible for reading the requested amount of bytes, stopping incoming
121 data via the RTS line when the requested amount of data was received and
122 has to call the handler. By this, the HAL implementation can stay
123 generic, while requiring only three callbacks per HCI packet.
124 
125 ### H4 with eHCILL support
126 
127 With the standard H4 protocol interface, it is not possible for either
128 the host nor the baseband controller to enter a sleep mode. Besides the
129 official H5 protocol, various chip vendors came up with proprietary
130 solutions to this. The eHCILL support by Texas Instruments allows both
131 the host and the baseband controller to independently enter sleep mode
132 without loosing their synchronization with the HCI H4 Transport Layer.
133 In addition to the IRQ-driven block-wise RX and TX, eHCILL requires a
134 callback for CTS interrupts.
135 
136  void hal_uart_dma_set_cts_irq_handler(void(*cts_irq_handler)(void));
137  void hal_uart_dma_set_sleep(uint8_t sleep);

140 ## Persistent Storage API {#sec:persistentStoragePorting}
141 
142 On embedded systems there is no generic way to persist data like link
143 keys or remote device names, as every type of a device has its own
144 capabilities, particularities and limitations. The persistent storage
145 API provides an interface to implement concrete drivers for a particular
146 system. As an example and for testing purposes, BTstack provides the
147 memory-only implementation *remote_device_db_memory*. An
148 implementation has to conform to the interface in Listing [below](#lst:persistentDB).
149 
150 ~~~~ {#lst:persistentDB .c caption="{Persistent storage interface.}"}
151 
152  typedef struct {
153  // management
154  void (*open)();
155  void (*close)();
156 
157  // link key
158  int (*get_link_key)(bd_addr_t bd_addr, link_key_t link_key);
159  void (*put_link_key)(bd_addr_t bd_addr, link_key_t key);
160  void (*delete_link_key)(bd_addr_t bd_addr);
161 
162  // remote name
163  int (*get_name)(bd_addr_t bd_addr, device_name_t *device_name);
164  void(*put_name)(bd_addr_t bd_addr, device_name_t *device_name);
165  void(*delete_name)(bd_addr_t bd_addr);
166  } remote_device_db_t;
167 ~~~~
